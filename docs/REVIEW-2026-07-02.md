> Four parallel review agents (deltat core engine, deltat transport, tap SDK packages, tap apps),
> findings re-verified by hand against the code that actually ships: deltat **main @ 2f76d5a1**
> (the local feat/clock-seam worktree is fully merged and behind main) and tap **feat/demo-v2 @ 5e61029**.

## The meta-finding: branch drift is the root cause of half the majors

The same defect keeps appearing in different clothes: code on one branch assumes a kernel or
sibling that lives on another branch.

- This deltat worktree (feat/clock-seam) is behind main by PRs #7-#12. Reviews, comments, and
  SDK assumptions made against it are stale on arrival.
- The tap SDK comment in `rules.ts` says the kernel can't batch rule inserts. Main can (PR #8).
  The SDK still loops one INSERT + one group-commit per rule.
- `calendar/server.ts` is a near-copy of `demo/server.ts` frozen before demo got its hold-order
  fix, Zod validation, connection caps, and stream guard. Every one of those is now a calendar bug.
- The SDK ships a `schedules` class for a kernel feature that was removed entirely.

Cheap structural fixes: delete merged worktrees (`git worktree remove`), extract one shared WS
server module for the two apps, and pin the SDK to a kernel version it is actually tested against.

## Critical (fix before showing anyone)

### C1. calendar booking is broken end-to-end · `tap/calendar/server.ts:50-62`
`handleConfirm` calls `dt.bookings.create` first and `dt.holds.release` after. The engine treats
the client's own active hold as a conflict (`deltat/src/engine/conflict.rs:51-52`), so every
confirm fails. `demo/server.ts:87-90` documents exactly this and releases before booking; the
calendar copy predates that fix. Compounding it, `state.holdId` is nulled before the create, so
after the failure neither confirm nor close releases the hold: the slot stays blocked for the
full 5-minute expiry. Fix: mirror demo's order (release, then book) and null `holdId` only on success.

### C2. the live booking path has zero validation; the validated path is dead code · `tap/calendar/server.ts:21-43`
`booking-form.tsx` now books via the WS `confirm`, which holds and books raw `msg.start/msg.end`
with no slot-grid, open-hours, duration, or past/future check (the engine deliberately never
consults rules in `check_no_conflict`). The audited, validated `createPublicBooking`
(`calendar/app/actions/public.ts:47-72`) has no callers left. An anonymous visitor can book 3am,
book multi-day spans, or repeatedly hold a huge span to lock the whole capacity-1 calendar 5
minutes at a time. Fix: re-derive the legal slot set server-side in `handleInit` (reuse
`getPublicSlots`) before `holds.place`, and delete or rewire `createPublicBooking`.

### C3. distinct tenant names share one WAL file · `deltat/src/tenant.rs:31-79`
The tenant map is keyed by the **raw** database name, but the WAL path uses the **sanitized**
name. `"prod"`, `"prod!"`, and `"pr.od"` become three separate Engines all backed by `prod.wal`:
each replays the shared file (full read access to the other tenant's data) and appends to it
concurrently; each engine's compactor rewrites via tmp-file rename, unlinking the inode the other
engine's `BufWriter` still holds, silently losing its subsequent events on restart. Fix: sanitize
before lookup and key the DashMap by `safe_name`.

Same file, same class: `get_or_create` is check-then-insert, so two racing first connections
build two Engines on one WAL path and both spawn the reaper/compactor/GC triple; the loser's
tasks hold its Arc alive forever (`tenant.rs:31-77`). Fix: DashMap `entry()` API, spawn
background tasks only for the winner.

### C4. remote panic via negative availability window · `deltat/src/engine/queries.rs:84-95`
`compute_availability` guards `end <= start` and window width but not sign. `start=-5000,
end=-1000` reaches `Span::new(query.start.saturating_sub(buffer).max(0), query.end)` =
`Span::new(0, -1000)`, which asserts (`model.rs:19`) and kills the connection task. `parse_i64_expr`
explicitly parses unary minus and wire passes availability bounds raw, so this is reachable from
SQL (probe-verified on main). Exactly the T-01/SEC-09 class believed closed. Fix: clamp query
bounds to `[MIN_VALID_TIMESTAMP_MS, MAX_VALID_TIMESTAMP_MS]` in `compute_availability`; the
multi and getMany paths route through it, so one guard covers all three.

## Major

### M1. SDK time-window filters are silently ignored by the kernel · `tap/packages/client/src/bookings.ts:61-67`, `holds.ts:36-42`
The SDK emits `WHERE resource_id = $1 AND start < $2 AND "end" > $3`, but the kernel's
`extract_resource_id_filter` only looks for `resource_id` and discards every other predicate, and
`Command::SelectBookings`/`SelectHolds` carry only `resource_id` (confirmed on main). Callers get
all rows, unwindowed, no error. Live consequence: `demo/lib/booking-flow.ts:33` uses
`holds.get(seatId, {start, end})` as an overlap check, so any non-overlapping hold on the seat
causes a false conflict. The ICS route and calendar `getWeekData` windows also do nothing.
Fix: filter client-side in `get()` after fetch, or add span filtering to the kernel commands.

### M2. partial resource update wipes unspecified fields · `tap/packages/client/src/resources.ts:26-55`
The SDK offers a partial-update signature, but the kernel defaults absent assignments
(`capacity.unwrap_or(1)`, name None, buffer None; `sql.rs:393-398` on main) and
`Event::ResourceUpdated` overwrites all three unconditionally (`store.rs:181-183`). So
`resources.update(id, { bufferAfter })` resets capacity to 1 and clears the name. Fix: the SDK
should always send all three fields (read-modify-write or required fields); the partial-PATCH
shape is a lie against this kernel.

### M3. SDK `schedules` class targets a removed kernel feature · `tap/packages/client/src/schedules.ts`
`INSERT/SELECT/DELETE ... schedules` hits `UnknownTable` on every deltat branch; the
`ScheduleSet`/`ScheduleRemoved` event variants can never be emitted. It is exported public SDK
surface that throws at runtime. Fix: delete the class, the type, and the event variants
(recurrence is expanded into Rules at the edge now, per MODEL-11/EDGE-03).

### M4. buffer conflict check is order-dependent · `deltat/src/engine/conflict.rs:66-76`
The check extends only the existing allocation's end by `buffer_after`, never the candidate's.
Probe-verified on main with buffer=30m, capacity 1: book A [10:00,11:00) then B [11:15,11:45)
is rejected; B then A is accepted (B sits inside A's cleaning window); the same pair as one batch
is rejected (the batch pairwise check is symmetric). Three admission paths, two semantics, and
admission depends on arrival order. Fix: pick one semantics. Either make the single-booking check
symmetric (candidate end + buffer vs existing spans, then reconcile `availability()` and
verify.rs), or declare the asymmetric point-semantics canonical and relax the batch checks to match.

### M5. ABBA deadlock between ancestor walks and batch lock ordering · `deltat/src/engine/queries.rs:43,97` + `mutations.rs:289-330`
`compute_availability` (and `add_rule`) holds the queried resource's guard while awaiting each
ancestor's read lock, walking **upward**. `batch_confirm_bookings` acquires write locks in
sorted-Ulid order, which for parent+child created in order is **downward**. A batch containing an
ancestor and a descendant (nothing forbids it: stadium GA section + seat), concurrent with any
availability query on the descendant, deadlocks permanently, and tokio's fair RwLock then wedges
every later request on both resources; the reaper skips them via try_lock forever. Fix: snapshot
`parent_id`, drop the guard, collect inherited rules, then re-lock (there is no snapshot isolation
across the walk today anyway), or reject ancestor-descendant pairs in one batch before locking.

### M6. day navigation broken west of UTC · `tap/calendar/components/slot-picker.tsx:50-60`
`prevDay`/`nextDay` do `new Date("YYYY-MM-DD")` (parsed as UTC midnight) then local-tz
`getDate`/`setDate`. In UTC-negative zones "next" is a no-op and "prev" skips a day. Line 66
already parses correctly with `date + "T00:00:00"`; use that in both handlers.

## Moderate

**deltat engine**
- **Watermark race loses a concurrent hold's expiry** · `mutations.rs:422-457`. `collect_expired_holds`
  ends with an unconditional store of the recomputed bound; a `place_hold` between scan and store
  has its `fetch_min` overwritten (worst case by `i64::MAX`). The hold expires but is never reaped
  until something lowers the watermark; its `HoldReleased` NOTIFY is silently lost. A value-CAS is
  not sufficient (fetch_min at equal values changes nothing); use a generation counter bumped by
  `place_hold`.
- **Batch commit: partial durability + N serialized fsyncs under all locks** · `mutations.rs:367-377`.
  Phase 2 awaits one `wal_append` per booking: a mid-batch WAL error leaves earlier bookings
  committed (contradicting AVAIL-06's all-or-nothing claim), and a full batch is up to 1,000
  sequential fsyncs while holding every batch resource's write lock. `wal_append_atomic` already
  exists; use it: one command, one fsync, one failure point.
- **`notify_ancestors` truncates the chain under contention** · `mod.rs:301-309`. The walk uses
  `try_read()` per ancestor while write guards are held across fsync awaits, so ms-long windows
  silently drop root-channel events (the demos subscribe at the root). Same pattern under-counts
  the depth check in `create_resource`. `parent_id` is immutable after creation: keep a lock-free
  parent index in `InMemoryStore` and make all ancestor walks lock-free (also shrinks M5's cycle surface).
- **`delete_resource` leaks `entity_to_resource` entries** · `mutations.rs:89-118`. Interval ids of a
  deleted resource are never unmapped; live-process memory grows with churn and stale ids resolve
  to misleading NotFound errors. Unmap while the guard is held.
- **`update_rule` bypasses invariants; entity ops ignore kind** · `mutations.rs:409-420` + `store.rs:116-134`.
  UPDATE can move a child's open window outside the parent's coverage (add_rule enforces AVAIL-09,
  update_rule doesn't). `resolve_entity_write` resolves any id regardless of kind:
  `cancel_booking(rule_id)` deletes a rule, `update_rule(booking_id)` morphs a booking into a rule.
  Duplicate caller-supplied ids silently create twin intervals and repoint the index. `commit_hold`
  already shows the intended pattern (verifies the entity is a Hold); do the same everywhere.

**deltat transport**
- **LISTEN on arbitrary ULIDs permanently leaks broadcast channels** · `wire.rs:911-926` + `notify.rs:28-34`.
  Subscribe never checks the resource exists and `NotifyHub::remove` runs only on real deletes, so
  reconnect-and-LISTEN cycles grow tenant memory without bound (each entry pre-allocates a 256-slot
  ring). Reject LISTEN for nonexistent resources and/or reclaim entries at `receiver_count() == 0`.
- **Dead forwarder handles block re-LISTEN** · `wire.rs:911-916`. After a resource delete the forwarder
  exits, but its JoinHandle stays in the per-connection map: a later LISTEN on the same id no-ops
  (delete-then-recreate leaves the client deaf) and dead entries count against the 100-subscription
  cap. `retain(|_, h| !h.is_finished())` before the checks.
- **MAX_QUERY_LEN enforced only at Execute** · `wire.rs:562-572`. Parse accepts up to pgwire's ~1 GiB
  packet limit and Describe runs `count_params` (4x char materialization) plus a full sqlparser
  parse before the 1 MB check ever runs. Return error 54000 from `parse_sql` itself.
- **Multi-statement simple query silently drops all but the first** · `sql.rs:30`. `"INSERT ...; INSERT ...;"`
  executes statement one and reports success. Return `Unsupported` when `stmts.len() > 1`.
- **rules/holds INSERT accept a column list but ignore it** · `sql.rs:62-104`. Both ids are ULIDs, so a
  reordered column list silently lands the rule on the wrong resource. resources/bookings are
  column-aware; extend the same mapping or reject non-canonical column orders.

**tap SDK**
- **`getMany` doesn't chunk** · `bookings.ts:76-91`, `holds.ts:51-66`. More than MAX_IN_CLAUSE_IDS
  (200 release) ids is a hard parser error. Chunk and merge.
- **Empty-string label diverges** · `bookings.ts:28`. `if (b.label)` treats `""` as absent on the
  single-row path (stores NULL, returns `label: ""`) while the batch path stores `""`. Use `!= null`.
- **Event callback errors swallowed** · `events.ts:16-22`. `callback(event)` sits inside the JSON-parse
  try, so subscriber exceptions vanish. Parse inside try, invoke outside.
- **`rules.create` still loops one INSERT per rule** · `rules.ts:29-32`. The comment claims the kernel
  can't batch; main merged `BatchInsertRules` (PR #8). Emit one multi-row INSERT (one group-commit).

**tap apps**
- **Any client's HoldPlaced flips "held for you"** · `calendar/hooks/use-websocket.ts:93-95`,
  `demo/hooks/use-websocket.ts:148-150`. The hook sets `connected = true` on any `HoldPlaced` event
  without matching span/id, so another visitor's hold re-enables this client's Book button after
  its own hold failed. Match the event span, or send an explicit `{type:"held", id}` ack.
- **calendar/server.ts regressed demo's hardening**. Untrusted `msg: any` (demo has a Zod
  discriminated union), two banned non-null assertions (`parse(req.url!, ...)` lines 82, 88), no
  connection caps, no stream age guard. Share one WS server module; the copy is how this happened.
- **WS init lifecycle races** · `demo/server.ts:147-165`, `calendar/server.ts:96-120`. (a) close-during-init:
  the close handler can run while `handleInit` still awaits, leaking the pg LISTEN and orphaning the
  hold until expiry; after init resolves, check `readyState` and clean up if closed. (b)
  `initialized = true` is set before the awaited init, so the catch's close-if-uninitialized can
  never fire; set it only on success.

## Verified clean

SQL injection in the SDK (all values via $N params or tagged templates, identifiers hardcoded);
recurrence expansion incl. DST and week boundaries; the no-client-side-availability rule holds in
both apps; WAL replay divergence (replay and live share `apply_event`); hold-expiry boundary
consistency across availability/conflict/reaper/GC; TLS loading; graceful shutdown and permit
accounting; the availability Describe schema design.

## Suggested order of attack

1. **C1 + C2** calendar booking order + WS-path validation (broken and abusable today, user-facing).
2. **C3** tenant keying + entry() race (data corruption across tenants, cheap fix).
3. **C4** negative-window guard (remote panic, one clamp).
4. **M1 + M2** SDK window filter + partial update (silently wrong or destroyed data on live paths).
5. **M4** decide the buffer semantics, then align all three admission paths.
6. **M5** parent index + drop-guard-before-ancestor-walk (also fixes the notify truncation).
7. Batch atomic append, watermark generation counter, LISTEN leaks, then the minors opportunistically.
