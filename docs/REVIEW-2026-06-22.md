# deltat + tap — Final OSS-Readiness Review

**Date:** 2026-06-22 · **Reviewer:** Lead reviewer (multi-agent: 7 area reviewers → adversarial verification → synthesis)
**Inputs:** verified per-area findings (engine core, transport, runtime/storage, tests/CI, SDK, calendar app, demo app), `docs/REQUIREMENTS.md` (authoritative), prior `tap/AUDIT.md` (2026-06-20).

> Method: each area was reviewed against the project's coding principles, an OSS-maintainer "is this representable to the public domain?" lens, and the REQUIREMENTS spec; every finding was then adversarially re-checked against HEAD code and dropped if not concretely confirmable. Branches: deltat `feat/clock-seam`, tap `feat/demo-v2`.

## 1. Executive Summary

**deltat (the Rust kernel) is representable to the public domain after one focused hardening pass.** The core is strong and the most important prior-audit items are now fixed: the untrusted-span panic (SEC-09/T-01) and the `buffer_after`/`expires_at` overflow DoS (prior audit's S4) are closed at HEAD; the `Command` seam is cleanly extracted into a transport-neutral `src/command.rs` (prior audit's highest-leverage refactor, item #5); the read/write availability contract is property-tested against a brute-force reference (2000 cases × 3 blocks). What remains for deltat is a small number of reachable panics on untrusted/concurrent paths the prior pass did not cover.

**deltat's single biggest blocker:** the availability query-window subtraction (`src/engine/queries.rs:89` and `:127`) panics on `i64` overflow from untrusted SQL bounds, with `overflow-checks = true` in release — same *class* as the DoS just fixed, but on the read path, reachable from any authenticated tenant. One validation + one regression test from closed (ENG-1).

**tap (apps + SDK) is NOT representable yet — it has a true blocker.** The `@open-tap/client` SDK is clean (no `as any`, no `!`, every `sql.unsafe` parameter-bound) and the prior audit's WebSocket finding (AUDIT.md S3, Security D) is genuinely fixed *in `demo/server.ts`*. But the fix was never ported to `calendar/server.ts`, the live public booking path for the real product, which still takes untrusted JSON as `any`, validates nothing, uses `req.url!`, and books before releasing the hold.

**tap's single biggest blocker:** `calendar/server.ts` (CAL-1) — the live `/book/[slug]` WebSocket path is unvalidated, untyped, and untracked in git, while the hardened `createPublicBooking` that should run is dead code nothing calls (CAL-3). The validated path is unused and the live path is unvalidated.

**Bottom line:** deltat can ship publicly once ENG-1 (ideally also RT-1, ENG-6, RT-6) is fixed and README/spec staleness is corrected. tap should NOT ship until CAL-1/CAL-3 are reconciled, the hardcoded production secrets (CAL-6) are gated, and the dead `Schedules` surface (SDK-1) is removed — each would actively mislead or endanger a public adopter.

## 3. Findings by Severity (duplicates merged)

### BLOCKER

**B1 — calendar/server.ts: live public booking WS takes untrusted JSON as `any`, validates nothing, books before releasing the hold** *(merges CAL-1 + CAL-2)*
`tap/calendar/server.ts:21,43,49-62,82,88,100-118` (untracked working-tree file, but the running code). The `/book/[slug]` → `useHoldWebSocket` → `/ws` path reads `msg.resourceId/start/end/label` off `JSON.parse` with no schema check and `req.url!` assertions — violating the project security rules (no `as any`/`!`/unauth client-trust) and PROTO-15. `demo/server.ts` was fixed for exactly this (Zod discriminated union, `json: unknown`, release-before-book) and the two files now byte-diverge. The confirm path books *before* releasing the client's own hold (CAL-2), so a real confirm rejects against the user's own hold (AVAIL-07; see `demo/server.ts:71-73`). **Fix:** port the demo fix verbatim, then delete one file and share one `server.ts` between demo and calendar; commit it.

### HIGH

**H1 — Availability query-window subtraction panics on i64 overflow from untrusted SQL (deltat's top blocker)** — ENG-1. `src/engine/queries.rs:89` and `:127`. `query_end - query_start` runs before any bounds check; `start`/`end` parsed straight from SQL (`sql.rs:184-202`), never validated on the query path (`wire.rs:213-253`). With `overflow-checks = true` (`Cargo.toml:39-40`) and per-connection `tokio::spawn`, `start=-1, end=i64::MAX` panics the task (runtime-confirmed, exit 101). Violates PRIN-08; partially negates SEC-09/AVAIL-17/T-01. **Fix:** validate bounds against `[MIN_VALID_TIMESTAMP_MS, MAX_VALID_TIMESTAMP_MS]` (or `checked_sub`/`saturating_sub`) before the width math in both functions; move the multi-path inverted/empty guard ahead of the subtraction; add an `i64::MIN/MAX`-through-the-wire regression test. (Minor: the "multi overflows for ANY inverted values" wording is overstated — `start=i64::MIN, end=-1` is clean — but overflow is reachable for many inputs.)

**H2 — Background WAL compaction panics on a contended resource lock** — RT-1. `src/engine/mutations.rs:402` `rs.try_read().expect("compact: uncontended read")`. Mutations hold the per-resource write lock across the awaited WAL append in `persist_and_apply` (e.g. `confirm_booking` mutations.rs:181→189); compaction runs in bare `tokio::spawn`'d GC/compactor tasks (`tenant.rs:63-75`) with no serialization and no `catch_unwind`, so a mid-mutation resource makes `try_read()` Err and `.expect()` panic — silently killing that tenant's compactor/GC, after which the WAL grows unbounded. Violates PRIN-08; contradicts ENG-13 under concurrency. **Fix:** skip a contended resource as `collect_expired_holds` (mutations.rs:331) / `gc_past_intervals` (mutations.rs:355) already do, or fall back to `rs.read().await`.

**H3 — Dead `Schedules` class issues INSERT/DELETE/SELECT against a kernel-removed `schedules` table** *(merges SDK-1 + SDK-2 + SDK-3)*. `tap/packages/client/src/schedules.ts:42-95`; registered at `client.ts:8,26,50`, `index.ts:10,29`; stale `ScheduleSet`/`ScheduleRemoved` variants at `types.ts:110-120` ("Matches deltat's Rust Event enum" at `types.ts:53` is now false). Kernel has no `schedules` table (MODEL-11, GAP-07/08); `Schedules.set()` throws at runtime against the live server; zero consumers (former caller already deleted). Biggest phantom-feature trap for a new contributor. **Fix:** delete the class + registration, the two Schedule `DeltaTEvent` variants, the `Schedule` type, the 4 mask/time helpers, and `__tests__/schedules.test.ts` — **preserving `localUtcOffsetMinutes` (live, `calendar/app/actions/setup.ts:34`) and `DayName` (live)**; relocate `localUtcOffsetMinutes` out of `schedules.ts` first. `expandRecurrence → rules.create` is the live path.

**H4 — createPublicBooking is fully validated but never called; the live path bypasses it** — CAL-3 (coupled to B1). `tap/calendar/app/actions/public.ts:47-72` re-derives open slots and asserts an exact match (`:64-66`) — where the prior audit's S2 fix landed — but a repo-wide grep finds only its own definition. The UI books over the WebSocket (B1) instead, with none of that validation. **Fix:** pick one path — route the UI through `createPublicBooking` and delete the WS confirm branch, or move its slot re-validation into the WS handler and delete the unused action.

**H5 — Orphaned duplicate TypeScript client (`deltat/client`) ships in the publishable footprint** — XPORT-1. `deltat/client/src/index.ts:1-408`; `README.md:196` names `@open-tap/client` as canonical. Zero importers; structurally different `DeltaT` (snake_case, no recurrence); carries the only forbidden `null!` in the area (`index.ts:95`). Violates "no duplicated state" + "no non-null assertions." **Fix:** delete `deltat/client`; `@open-tap/client` is canonical (vendor/submodule if needed). Removes the `null!`.

### MEDIUM

**M1 — Negative `min_available` wraps through usize into a garbage availability result** — XPORT-2. `sql.rs:256-259` (`v as usize`), `queries.rs:133,153,159`. `-1` → huge usize → passes `== 0` guard → `as i32` wraps negative → garbage sweep. Read-only but untrusted input producing nonsense (PRIN-08 spirit). **Fix:** reject negatives at the SQL boundary, or treat `< 0` as empty alongside `== 0`.

**M2 — Booking.label is a second descriptive String in the kernel** — ENG-2. `model.rs:61,189,223`; `store.rs:167`; `mutations.rs:188`. Never read by `availability()`/`check_no_conflict`; spends a limit (mutations.rs:174-177). Violates MODEL-07/08/09/NOT-02; the one unresolved purity exception (`name` grandfathered). **Fix:** replace with `external_ref: Ulid` (GAP-02/T-02). PII-relevant for a public append-only/federated log.

**M3 — Hold-expiry & conflict comparisons evaluate against steppable CLOCK_REALTIME** — ENG-3. `conflict.rs:51,115`; `availability.rs:49`; `mutations.rs:334,365`; `clock.rs:24-26`. A backward wall-clock step (NTP/negative leap second) can make an expired hold read as live. Violates HW-01/02/03 (GAP-11/T-11). **Fix:** split `now_wall`/`now_mono`, compare a monotonic elapsed-delta vs TTL, add HW-20 adversarial-clock tests (`TestClock.set()/advance()` already supports this).

**M4 — hold→book is non-atomic (no CommitHold) — release_hold then confirm_booking is a TOCTOU window** — ENG-4 (surfaces at app layer as CAL-2/B1). `mutations.rs:159-164` + `:166-190`; no `commit_hold` symbol. Violates AVAIL-07; reconfirmed by GAP-AUDIT. **Fix:** `CommitHold(hold_id)` under one lock excluding that hold; decide the hold-capability model first (SEC-03/T-06).

**M5 — Booking conflict-check ignores blocking rules while availability subtracts them** — ENG-5. `conflict.rs:38-73` (Hold/Booking only) vs `availability.rs:74-81` (subtracts blocking). A blocked window reads unavailable yet a direct book/hold succeeds. AVAIL-16/T-03 (decision pending). **Fix:** decide T-03 — reject blocked spans in `check_no_conflict` (parity) or document the asymmetry in code.

**M6 — Notify broadcast channels never removed on resource deletion (unbounded growth)** — RT-3. `notify.rs:46-49` vs `mutations.rs:70-92`. `delete_resource` never calls `notify.remove(id)`; the only caller is a unit test. A long-lived tenant leaks one `Sender` per ever-subscribed resource. "No over-engineering / one source of truth." **Fix:** call `notify.remove(&id)` in `delete_resource`, or delete the unused method.

**M7 — calendar saveSchedule dual-writes schedule.json AND deltat rules with no transactional tie** — CAL-4. `calendar/app/actions/setup.ts:36-65`; `calendar/lib/schedule-store.ts:32-37`. `getSchedule` reads only the JSON while availability comes from rules; a crash between writes drifts them. Violates PRIN-06. **Fix:** make deltat the single source (opaque per-resource metadata), or treat JSON as advisory and funnel both writes through one helper.

**M8 — Hardcoded default credentials and HMAC secret let calendar run in production with publicly-known values** — CAL-6. `calendar/lib/config.ts:2-4`; `calendar/lib/deltat.ts:8`. `pass ?? "changeme"`, `secret ?? "dev-secret-change-in-production!!"`, `password ?? "secret"` — the HMAC key is in the public repo, so any visitor can forge a `cal_session` cookie. No startup guard. **Fix:** fail fast in production if `CAL_SECRET`/`CAL_PASS`/`DELTAT_PASSWORD` are unset or equal the dev default; document required env vars.

**M9 — Password and HMAC comparisons are not constant-time (calendar)** — CAL-5. `calendar/app/actions/auth.ts:11`; `calendar/lib/auth.ts:23`. Short-circuit `!==` leaks timing; the network-observable HMAC compare is the meaningful side channel. **Fix:** `crypto.timingSafeEqual` on equal-length Buffers for the HMAC; hash-then-compare for the password.

**M10 — 13 byte-identical demo route shims the registry can generate** — DEMO-2 (count 14→13). `tap/demo/app/demos/{...}/page.tsx`. Same 8-line shim differing only in one id literal; `demo/app/embed/[example]/page.tsx:12-24` already collapses the pattern. DRY/KISS. **Fix:** one `app/demos/[example]/page.tsx` validating against `ALL_EXAMPLE_IDS` + id→component map + `generateStaticParams = enabledExampleIds()`.

**M11 — Stale spec/docs that contradict HEAD (merged BUILD-1 + BUILD-2 + DEMO-4 + RT-5)** — (a) `REQUIREMENTS.md:313` TEST-12 claims property/proptest deps absent, but `Cargo.toml:30` declares `proptest` + 3 `proptest!` blocks (2000 cases) pass — *contradicts the prior audit's "no property tests"*; (b) `README.md:182-194` architecture omits 8 of 15 src files incl. the clock-seam/multi-tenant/TLS layers; (c) root `README.md:52-61` demos table lists demos that don't exist, omits 7 that do; (d) `README.md:175-194` env table documents 4 of 10 env vars. REQUIREMENTS is authoritative, so a false spec line is the worst staleness. **Fix:** correct TEST-12; regenerate the architecture block from `src/`, the env table from `main.rs`/ENG-22, the demos table from `manifest.ts`.

**M12 — Hold→book race on the SAME resource is uncovered (test gap)** — BUILD-6. `src/engine/tests.rs:3868-3893`: the only concurrency test uses distinct resource ids (WAL fan-in, not contention). No test races a hold and a competing booking against one capacity-limited resource (TEST-12/TEST-17). **Fix:** add a deltat-side async test racing N concurrent bookings/holds against a capacity-1 resource on an overlapping span, asserting exactly capacity succeed.
