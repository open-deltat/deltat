# Auth & Payments: the edge security spec

> Companion to [`REQUIREMENTS.md`](REQUIREMENTS.md). Authoritative for **who may do what, how money moves, and where every secret lives.** Same rules as the forever spec: each line is a checkable requirement with a stable ID; statuses use the REQUIREMENTS legend (✅ 🟡 📋 ⏸ ❌ ❓). Where this doc and REQUIREMENTS conflict, REQUIREMENTS wins on kernel facts; this doc is authoritative on the auth/payment edge.
>
> **Fact-checked against HEAD (`feat/clock-seam`).** Verified directly: `command.rs` is transport-neutral (PROTO-10 seam exists); `auth.rs:18` ignores `LoginInfo` and returns `DELTAT_PASSWORD` for any user; `wire.rs:72` derives the tenant from the unauthenticated `database` connection metadata (PROTO-11); expiry math (`conflict.rs:51/115`, `availability.rs:49/602`) compares `expires_at` against `now` = `SystemClock` = `CLOCK_REALTIME` (`clock.rs:25`, a steppable source); there is **no** `commit_hold`/`CommitHold` symbol; the path is `release_hold` (`wire.rs:176`) then `confirm_booking` (`wire.rs:188`), a real TOCTOU; `get_holds` (`queries.rs:253`) returns the bare hold `Ulid` + span + `expires_at`.

## 0. The one invariant under everything

**PROTO-AUTH-00** ✅ (principle) **The kernel takes `(Command, idempotency Ulid, integer now)` and nothing else.** No secret, token, identity, price, ACL principal, or PII crosses into `src/engine`. Auth is a gate **above** `Command` dispatch; payment is a side-channel keyed by the kernel idempotency `Ulid` (PAY-01/PAY-05, MODEL-07/09, NOT-01/02). Verified: `command.rs` depends only on `model` + `ulid`. This invariant is why the entire auth/payment story is an **edge** story and `src/engine/*` needs zero change.

**PROTO-AUTH-01** 📋 **Secure-Write-First, Capability-Anchored (SWFCA).** One spine applied four times (MCP auth, cross-calendar, federation, payment), sequenced on the real `pgwire → PROTO-03` path so something secure ships **this week**, anchored on a single home keypair (distinct `kid` per job) that signs the hold capability and doubles as federation identity. Boundaries drawn now (free); crypto/federation code built only on the FED-08 trigger (expensive).

## 1. MCP / HTTP caller auth: Resource-Server-only

**PROTO-AUTH-02** 📋 **tap's MCP/HTTP surface (PROTO-03/04) is an OAuth 2.1 Resource Server, never a token issuer** (the 2025-06-18 spec split). Validate tokens; **delegate** issuance to a provider (WorkOS / Auth0 / Keycloak / Cloudflare `workers-oauth-provider`). ❌ **Do not hand-roll an Authorization Server in Rust**: the spec never requires it and it is where redirect/consent/PKCE bugs live.

**PROTO-AUTH-03** 📋 **Discovery + validation mechanics (multi-tenant phase).** WRITE tool with no token → `401` + `WWW-Authenticate` carrying a `resource_metadata` URL → serve RFC 9728 Protected Resource Metadata at `/.well-known/oauth-protected-resource` → client does RFC 8414 AS discovery, DCR (RFC 7591) or CIMD, PKCE (RFC 7636, mandatory), auth-code flow, and **MUST** send the RFC 8707 `resource` parameter = this node's canonical URI in **both** authorize and token requests. Every request carries `Authorization: Bearer` (never a query string, never logged, SEC-06). As RS, **MUST** validate signature **and** audience (`aud`/`resource` == canonical URI). Re-validate on **every** call; **MUST NOT** use the session/connection as the principal; bind any session id as `<user_id>:<session_id>` with a non-deterministic id.

**PROTO-AUTH-04** 📋 **Audience ≠ tenant.** Audience binding is **per-resource-server**; one node fronts **many** tenants behind **one** canonical URI. The audience check proves "minted for this node"; the **tenant** is derived from a **separate** claim: the PROTO-AUTH-08 credential→tenant binding. Missing/unmappable = reject. *(Correction: do not treat the node's single audience as a tenant name; that is the standards bug a naive audience=tenant collapse introduces.)*

**PROTO-AUTH-05** 📋 **Read/write asymmetry, with an open-read threat model.** `search_bookable`/`get_availability` MAY be lightly keyed (VIS-12; FED-09, the read is topology-free portable data). But they are **not** a free unbounded oracle:
- A resource is discoverable only if its publisher **opted in** (T-09); resource-ID enumeration (ULIDs are timestamp-prefixed and guessable) **MUST NOT** reveal unpublished resources.
- Anonymous/lightly-keyed reads get per-IP/per-key rate limits **and** per-query cost bounds (cap time-window width + IN-list size, mirroring `MAX_IN_CLAUSE_IDS`) so availability cannot be scraped as a real-time competitor occupancy feed (SEC-05).
- `book`/`hold`/`CommitHold` are **always** authenticated and attributable.
- `SelectHolds` is **owner/operator** data, not discovery data; it returns the bare hold `Ulid` (`queries.rs:253`). Scope it to a write/owner credential; never return hold ids cross-tenant or to anonymous callers. **Availability exposes free spans, never hold identifiers.**

**PROTO-AUTH-06** 📋 **Capability graft + structural no-passthrough.** A validated OAuth bearer is **not** carried inward. The edge exchanges it for a scoped, depth-bounded inward capability (delegation, not impersonation, record agent + user U). Enforcement is **structural**: the inward capability is the only thing dispatch accepts; the bearer is dropped at the runner boundary by construction (never on the `Command`, never into `execute_command`), so token passthrough is **impossible**, not merely forbidden. Any outbound peer/PSP call is a separate edge client minting its own credential, sharing no token store with the inbound gate.

**PROTO-AUTH-07** 📋 **Phasing + spec floor.** Single-operator demo ships as local stdio MCP (`--env` secret, process isolation) **or** remote HTTP behind one static bearer over TLS, honest for one operator, explicitly transitional. The full RS role lights up at the multi-tenant trigger and reuses PROTO-AUTH-08 verbatim. **Target 2025-06-18 as the floor; advertise the protocol version** (2025-03-26 server-as-AS and 2025-11-25 CIMD/OIDC deltas differ materially).

## 2. Replacing the shared-password hole: the P0 spine

**PROTO-AUTH-08** 📋 **P0a (this week): credential→tenant binding.** `DeltaTAuthSource` resolves a per-principal secret + the principal's allowed-tenant set; `resolve_engine` derives the tenant from the **authenticated principal**, never from connection metadata. **Remove the `database`-metadata path entirely**: missing binding = reject the connection; never a conditional fallback (multi-tenant rule: missing filter = reject, never return all). P0 gates **all** access (read **and** write) to the authenticated tenant, strictly more restrictive than today, so no regression. Bind the LISTEN/NOTIFY channel (PROTO-12) to the authenticated tenant too. Closes PROTO-11 / degraded SEC-01. *(The smallest secure diff is the tenant-derivation removal, not the hashing.)*

**PROTO-AUTH-09** 📋 **P0b (multi-tenant trigger): swap the config map for an argon2id-hashed credential store** keyed by username. Same binding semantics; secrets now safe at rest for multiple real users.

**PROTO-AUTH-10** 📋 **P1: read/write ACL scope (SEC-02).** A credential carries `{read | write[, resource-set]}`, enforced in `execute_command` (`wire.rs:104`) as a match on the `Command` variant **before** dispatch: `Select*` = read; `Insert/Update/Delete/Batch/Hold` = write. Ship the PROTO-AUTH-05 open-read rate limit + cost bounds in the same phase. Never expose a write-adjacent read path with only the ENG-24 connection `Semaphore`.

**PROTO-AUTH-11** 📋 **P2: transport-neutral runner + OAuth-RS bearer gate.** Carve the dispatch runner out of pgwire (PROTO-10). One token-validation gate above it (PROTO-AUTH-03/04). HTTP/JSON (PROTO-03) + MCP (PROTO-04) ride **one** runner; the ADAPTERS write-guard lives here **once**; the shared password never reaches PROTO-03/04.

## 3. Hold-as-capability and the clock prerequisite

**SEC-03** ❓→📋 **Resolved at the edge: the raw `hold_id` never authorizes commit.** `CommitHold` authorization is a home-**signed**, audience-bound, short-TTL, attenuable capability: `sign(hold_id, resource, span, expires_at, action=commit, [price], audience)`, public-key (biscuit/UCAN), never HMAC. **The kernel `CommitHold(hold_id)` is unchanged and unaware of the capability**: the edge runner verifies the signature + monotonic-clock expiry, then calls `CommitHold(hold_id)` only on success. The capability never crosses into `src/engine`; it is an edge artifact, exactly like the OAuth bearer (this keeps T-06's resolution in the SEC layer, not a MODEL change). *(A signed capability whose secret inputs are all published by an open read endpoint provides zero confidentiality; hence PROTO-AUTH-05 pulls `SelectHolds` off the open surface and the capability is returned only to the authenticated principal that placed the hold.)*

**AVAIL-07** 📋 **CommitHold MUST be a single-lock atomic transition** (one persisted `HoldCommitted` event) converting hold→booking with **no** release-then-reinsert gap, excluding only that hold from the conflict check. *(A signed capability authorizes **who** may commit; it does **not** make the commit atomic. Today's `release_hold`→`confirm_booking` is a real TOCTOU: a racing/retrying caller can win the transient free window.)* Verify with a concurrency test: two `CommitHold` racers + one independent `confirm` racer on one span → exactly one winner, no transient free window.

**HW-01 / GAP-11** 📋 **Monotonic-expiry is a hard prerequisite of CommitHold, not later work.** Split `Clock` into a wall source (stored/external instants) and a monotonic source (all elapsed/expiry/conflict math); compare a monotonic delta-vs-TTL. Today `now` = `CLOCK_REALTIME` (`clock.rs:25`), a backward NTP step or post-2035 negative leap second makes an expired hold (and an expired short-TTL **capability**) read as **live**. Land HW-01/HW-02/HW-03 + the **HW-20** adversarial backward/stalled-clock tests (asserting an expired hold **and** an expired capability both stay expired across a backward jump) **before** any capability ships. CommitHold is unsafe on a steppable clock even with a raw `Ulid`.

## 4. Cross-calendar bridging (deferred subsystem)

> Gated on the first user who asks to sync an external calendar, **not** on the critical path to first-secure-write + first-deposit. One sentence constrains P0-P3: external-calendar tokens live in an edge Credential store, encrypted at rest, never in the kernel; the kernel sees only a booking `Ulid` + opaque `external_ref`.

**PROTO-AUTH-12** ⏸ **Outbound (deltat → Google/Graph/iCloud).** deltat is an OAuth/CalDAV **client**. Adopt Cal.com's shape: a `Credential` keyed by `(tenant, user, provider)` with an **AES-GCM**-encrypted token blob (KMS/HSM keys, datastore off the public internet, refresh-token rotation + reuse-detection), plus a `BookingReference {external_event_id, externalCalendarId, credentialId, deltat_booking_id}` so cancel/reschedule propagate **and** a round-tripped event is recognized by its external id (kills the echo/double-book loop). **Scope is per-direction**. Least privilege falls out of deltat's read/write split: read = Google `calendar.freebusy` / Graph `Calendars.Read` (`.Read.Shared` for `findMeetingTimes`); write = Google `calendar.events` / Graph `Calendars.ReadWrite`, requested **incrementally** on the write path. Never request Google's broad `.../auth/calendar`. Prefer **per-user delegated OAuth** (blast radius = one user) over Google domain-wide delegation / MS application permissions; if DWD is unavoidable, minimize API scopes + restrict the mailbox set + KMS the SA key. iCloud/Fastmail: app-specific passwords, Basic-over-TLS only (RFC 4791), high-value secrets, encrypted at rest.

**PROTO-AUTH-13** ⏸ **Inbound (external client → deltat-as-CalDAV / iTIP).** Forces a **per-principal** credential (RFC 4791/6764 `current-user-principal` → `calendar-home-set`) mapped to a deltat principal with a resource ACL (SEC-02), **never** the shared password. The **iMIP email path (RFC 6047) is DISABLED BY DEFAULT** and enabled only per-deployment with a defined authenticated channel (S/MIME where available, or a same-server authenticated submission path), **never** an envelope-from / From-header allowlist *(S/MIME is near-undeployable in practice, so the allowlist temptation is the real spoof vector)*. Defense = RFC 6638 **server-decides-accept**: deltat's engine is the **sole** authority and runs **both** the conflict/capacity check **and** the full attendee-scheduling-object check (ORGANIZER consistent, ORGANIZER ≠ the resource's calendar-user-address, an ATTENDEE matches the resource). A booking is real only after the home engine **authorizes** (authenticated originator + authorization) **and commits**, never because an email said so. The kernel capacity/window check is **not** an identity check. CalDAV-adapter code-review checklist item.

**PROTO-AUTH-14** ⏸ **GAP-02 label hazard.** Until `label: String → external_ref: Ulid` lands, the kernel's existing `Booking.label` is a free-text field that **can** carry an external title/PII. The cross-calendar adapter **MUST** pass `None` / an opaque `Ulid` only and **MUST NOT** populate `label` with any external event title (SEC-04/MODEL-09).

## 5. Federation auth (deferred until FED-08)

**FED-04** ⏸ **One home keypair, two jobs: distinct `kid` per job.** Each node is a sovereign origin with an Ed25519 keypair at its **own** `.well-known`; a peer verifies by fetching the issuer's key from the issuer's **own** domain and checking locally (the email-DKIM / ActivityPub / Matrix / OpenID-Federation shape, no central CA). The same key signs (a) s2s federation requests, (b) the hold capability, (c) the booking receipt, but use a **distinct signing key / `kid`** for capabilities vs federation identity, and publish a **keys array** at `.well-known` (Matrix `verify_keys` shape) with an overlap window for rotation, so a capability-key rotation does not reset server identity and vice versa. *(Mitigates the one-keypair blast-radius: a single key for everything makes a compromise forge federation identity, mint commit capabilities, and forge receipts simultaneously.)*

**FED-05** ⏸ **Capability revocation = TTL + fencing epoch.** Short TTL is necessary but not sufficient; a per-resource **epoch** (epoch + per-resource sequence + per-op nonce) means a rotation bumps the epoch and instantly invalidates outstanding capabilities. Document the rotation procedure (overlap, old-key grace, epoch bump), not a TODO.

**FED-AUTH-01** ⏸ **Public-key capabilities only** (biscuit/UCAN), never HMAC macaroons: in open federation no home trusts another with a shared root secret and an HMAC verifier can forge. **Sign the durable artifact**, not just the request (ActivityPub Object-Integrity-Proof lesson): the signed hold capability and booking receipt are portable objects, independently verifiable later for FED-03 Try-Confirm-Cancel dispute evidence.

**FED-AUTH-02** ⏸ **Cross-org = audience validation; no passthrough.** A capability minted for node A **MUST** be rejected by node B; each hop mints a fresh scoped credential (PROTO-AUTH-06 structural enforcement). Make per-origin `.well-known` the default, cheapest discovery path so federation does not re-centralize the way Matrix collapsed onto matrix.org; the indexer (FED-06) is a stale discovery **hint** only (T-08), never a trust root or commit point.

**FED-AUTH-03** ⏸ **Delegation, not impersonation; bound depth + scope.** Carry an actor/delegation claim so the home records agent + user U on every commit; bound delegation depth and scope **inside** the capability and reject unbounded sub-delegated tokens.

**FED-07** ⏸ **The saga stays unsolved.** Capability+payment composition does **not** make a cross-node hold+pay atomic: one commit + one failure is still possible. Payment must be reversible (capture only **after** commit, else refund), Try-Confirm-Cancel, idempotency-keyed on the kernel `Ulid`.

**NOT-05** ❌ **Build none of §5 until a real second operator exists (FED-08).** The single-operator `CommitHold` needs only a **local** key (audience = this node), monotonic-clock verify, no `.well-known` publication, no peer fetch, no cross-org audience (call this P4a). The federation job (key publication, peer verification, cross-org audience) is P4b on trigger. Library/RFC choices (biscuit vs UCAN, exact token-exchange mechanics) are **non-binding** until then, decided against the real second operator's constraints. The framed `Command` (PROTO-01) carries the capability metadata at the edge so the format stays swappable and never touches the kernel.

## 6. Payments

**PAY-x402-01** ⏸ **x402: YES to pay-to-USE the federation, NO as the deposit engine.** Use it as **one swappable settlement adapter** (deferred, last) for per-call discovery/search access (VIS-12): agent hits search → `402` → signs a gasless EIP-3009 `transferWithAuthorization` (USDC/EURC, one-time nonce, EIP-712 domain-separated) → non-custodial facilitator verifies+settles → `200` + receipt. Composes because x402 separates payment from identity: payment-settled and authz-resolved are independent gates (PAY-01/PAY-05).

**PAY-x402-02** ⏸ **Version-pin x402 v2** (as we version-pin MCP): headers `PAYMENT-REQUIRED` / `PAYMENT-SIGNATURE` / `PAYMENT-RESPONSE`; CAIP-2 networks (`eip155:8453` Base mainnet, `eip155:84532` Base Sepolia); `maxAmountRequired` → `amount`; the token's EIP-712 domain in `extra`. Advertise the version the adapter speaks. *(The v1 `X-PAYMENT` form is deprecated.)*

**PAY-x402-03** ⏸ **Adapter replay / double-settle guards.** (1) Idempotency-key the settle on the kernel `Ulid` (PAY-01) so a retry after a downstream failure cannot double-settle irreversible USDC. (2) Verify authz **before** settle, never in parallel, never hold un-authorized irreversible funds. (3) Pin/allowlist the facilitator and reject server-directed recipient changes (defend v2 dynamic-routing recipient manipulation). (4) Bind the x402 nonce/EIP-712 domain to the hold capability so a captured authorization cannot be replayed against a different hold. (5) Keep x402 strictly **off** any refund-requiring path; for access-only, a settled payment with a failing downstream commit yields a defined non-custodial outcome (the call fails; no funds owed back). Assert this with a test.

**PAY-01 / PAY-02 / PAY-04 (VIS-08)** 📋 **Deposits flow over Stripe manual-capture (P3, boring rails first).** The HOLD is modeled in deltat's own kernel (it already has first-class holds with auto-expiry, the hold **is** authorize-and-hold) and money settles against Stripe `PaymentIntent(capture_method=manual)` via Stripe Connect **direct charges** (PAY-04, the protocol never custodies; resource PSP = merchant of record). `place hold → PaymentIntent(manual)` keyed by the hold `Ulid`; `CommitHold` → capture; expiry/cancel → void; no-show → capture the fee. **Capture only after commit** (FED-07; payment is a fallible, reversible, idempotency-keyed step, INV-06). *(x402's "exact" scheme is immediate and irreversible: no hold, no capture-later, no partial capture, no refund, so betting deposits on it reintroduces the custody problem PAY-04 forbids; adoption is also thin/meme-coin-inflated.)* Money sits behind a settlement-adapter abstraction so x402 is one adapter among several and the capability stays rail-agnostic (it states price + audience + action; whether Stripe or x402 settles is an edge decision keyed by the same `Ulid`).

## 8. Open decisions

- **P0a store shape**: config/env map (recommended first diff) vs argon2id now. The security-critical change is removing metadata-derived tenant, not hashing.
- **Delegated AS provider**: WorkOS / Auth0 / Keycloak / Cloudflare. Vendor + cost, not design.
- **CalDAV inbound**: support it near-term or outbound-only first (inbound forces the full per-principal store + RFC 6638 authorization).

## 9. Ideal end-state: federation-native capability auth

> **Status: recorded destination, not a mandate.** This section is the worked-out end-state for the SEC-03 / FED-AUTH-01 / §8 *capability-format* open decision. It is **non-binding until FED-08** (a real second operator) **and** the first real multi-hop-delegation requirement, decided against their constraints, per NOT-05. It refines, and does not override, FED-04/05, FED-AUTH-01/02/03, FED-07, FED-09, SEC-03, AVAIL-07/08, HW-01/02/03, MODEL-12. Built on the SWFCA spine (PROTO-AUTH-00..11). The spec text elsewhere says *biscuit/UCAN*; this records **PASETO-v4.public core + Biscuit attenuation** as the **leading candidate**, with the reasoning, not as a spec change.

**FED-AUTH-04** 📋 (candidate) **Graduated public-key capability format, both Ed25519, one verify interface.** PRIMARY = **PASETO v4.public** for (a) the hold commit-capability (SEC-03), (b) the booking receipt (FED-AUTH-01 durable artifact), (c) every single-hop and cross-home-fresh-mint path. The algorithm is fixed by the literal `v4.public` version string: no `alg` header, so the JWT `alg:none` / RS256↔HS256 / kid-injection class is gone **by construction**. SECONDARY = **Biscuit** (Ed25519 chain + Datalog) **only** as the offline attenuation envelope on multi-hop sub-delegation (the one path PASETO cannot serve: narrowing a flat token needs a re-mint from the online home). REJECTED: macaroons (HMAC verifier needs the issuer's secret, fatal cross-home, FED-AUTH-01); raw JWT (alg-confusion); UCAN (same N-verify cost as Biscuit, drags DID/IPLD the hot path does not need). Signed CWT/COSE is the binary drop-in for the core if a CBOR transport ever dominates.

**FED-AUTH-05** 📋 (candidate) **Typed, origin-scoped keys: close the confused-deputy/substitution class.** A signed `typ` discriminator (`commit-cap | receipt | root`) is bound into the PASETO **implicit-assertion** and the Biscuit **root fact**, and checked **first**, so a receipt can never replay as a commit-cap under the shared FED-04 capability key. Each `kid` carries a `use:cap | use:identity` tag (FED-04 already mandates distinct kids). The verifier derives the **origin from a signed field**, then resolves the kid **only within that origin's `use:cap` keyset**: an identity-signed blob cannot verify as a capability, and cross-issuer kid substitution / cross-audience misses are structurally impossible. The origin **identity** key is pinned out-of-band on first contact and signs the `.well-known` key doc.

**FED-AUTH-06** 📋 (candidate) **Hot-path verify: verify-then-lock, zero network I/O.** Per commit at the edge runner: (1) dispatch on the leading version tag; ambiguous ⇒ fail **closed** (no downgrade). (2) Check `typ==commit-cap` first (FED-AUTH-05). (3) Resolve the `use:cap` kid within the signed origin's keyset; kid-miss fails closed + **background** refill (never a synchronous hot-path fetch, DoS guard). (4) ONE `verify_strict` (~28 µs AVX2 / ~41 µs serial, ed25519-dalek; rejects malleable sigs). Biscuit: `d`-block chain = `d` verifies + an explicit-allow Datalog authorizer. (5) Bind-checks as integer/string compares, all reading **edge** caches (never kernel state, never under the resource lock): `aud == canonical URI` (FED-AUTH-02); `action==commit`; resource-set contains target; nonce/epoch **claims == the edge fencing store's stored values** (FED-AUTH-08); a **hard tenant compare** (outside the Datalog authorizer) of the token's tenant claim against the tenant from the PROTO-AUTH-08 credential→tenant binding, missing/unmappable = reject, **never** connection metadata (PROTO-AUTH-04). (6) Expiry per FED-AUTH-09. **All of (1)-(6) complete before the per-resource write lock is taken**: a slow Biscuit verify must not serialize unrelated commits. (7) Only on success, under the lock, call the unchanged capability-unaware kernel `CommitHold(hold_id)` (AVAIL-07). The commit-capability is re-verified **every** commit (PROTO-AUTH-03) and **cannot** be amortized to handshake/batch; only the OAuth bearer is verified once-per-connection. Budget: common path ~28-50 µs, well under 1 ms but honestly a **~20-35 % tax on a group-commit-amortized durable commit** (~140 µs/commit at the HW-07 7k-txn/s target), single-digit-percent of a lone fsync (HW-06). Biscuit worst-case at depth × non-trivial caveats is a **~1 ms-class** full-verify (biscuit-rust's own figure). **Benchmark the real caveat rule-set on the deploy target (TEST-11/HW-07) before shipping**, and set the depth cap from measurement (start at **3**).

**FED-AUTH-07** 📋 (candidate) **Format follows the root; the home decides at consent.** `place_hold` returns a **capability**, not a bare `Ulid` (SEC-03/T-06; the raw `hold_id` is a non-authorizer). The hold-commit capability is a **flat PASETO** when the principal acts directly/single-hop (the common case, re-mint from the online home is one cheap verify); it is a **Biscuit** only when the human's consented scope explicitly permits sub-delegation, so an attenuable token never exists without an explicit human grant. Single-hop narrowing = **re-minting** a tighter flat token from the online home (still one verify). Multi-hop offline narrowing = Biscuit attenuation blocks, each **strictly nested** in its parent (monotonic), **depth- and scope-bounded** (FED-AUTH-03). **HOP depth (cross-home) and DELEGATION depth (user→agent→sub-agent) are orthogonal**: each cross-home hop mints fresh (FED-AUTH-02 no passthrough) so the cross-home path is depth-0 **by construction** and never needs Biscuit, while a Biscuit delegated token may still be presented to a home whose own cross-home mint is flat.

**FED-AUTH-08** 📋 (candidate) **The fencing nonce/epoch live in an explicit EDGE store, not the kernel.** PROTO-AUTH-00 forbids a nonce/epoch field in `src/engine` (absent from `model.rs`; `IntervalKind::Hold` carries only `expires_at`; the MODEL-11 event vocabulary is frozen). A per-tenant **edge fencing store** (`hold_id → {nonce, epoch_at_mint}`, `resource_id → current_epoch`) is the **authoritative source** the FED-AUTH-06 bind-check reads: the token carries the values as **claims**, the store holds the **truth** (comparing a token against its own field is circular). It is **durable** (its own small append-only log, or co-located with but **outside** the WAL) and **reconstructed on restart**. Revocation (FED-05) = mutate the stored epoch ⇒ every outstanding capability fails the cached integer compare. Edge-store/kernel disagreement after a partial crash ⇒ **fail closed**.

**FED-AUTH-09** 📋 (candidate) **Wall is the durable/portable expiry reference; monotonic is a same-process anti-backstep floor.** `CLOCK_MONOTONIC` **does not survive a process restart** (per-process, resets to ~0 at boot), and deltat is a WAL-replay system where restart is first-class, so the durable reference **must** be the persisted **wall** deadline: `HoldPlaced.expires_at` and the capability's `wall_deadline`, **both AVAIL-08 authority-assigned** (the home sets both so they cannot drift; this promotes **AVAIL-08 to a hard prerequisite** alongside HW-01/02/03+HW-20 and AVAIL-07). The local check: at process start capture `(anchor_wall, anchor_mono)`; `effective_now = max(now_wall, anchor_wall + (now_mono − anchor_mono))`, re-anchored **forward only** per read; expiry = `effective_now ≥ wall_deadline`. This **survives restart** (re-anchors from a fresh wall read, the best post-restart truth) **and** defeats an in-process backward NTP step (the monotonic term floors it). Pin `CLOCK_MONOTONIC` (vDSO-fast, NTP-slewed), **never** `CLOCK_MONOTONIC_RAW`. The signed artifact's expiry is **absolute wall time** (portable, third-party-verifiable for FED-03 dispute evidence and post-restart self-verification); the local floor merely refuses to honor it past `effective_now`. **HW-20 gains a restart case**: place a hold, persist + replay (simulating restart) under a clock that jumped backward between processes, assert the hold **and** the capability both stay expired.

**FED-AUTH-10** 📋 (candidate) **Decoupled, short commit-TTL.** `wall_deadline = now + min(SHORT_CEILING, hold.expires_at − now)`, `SHORT_CEILING` a named config (default **120 s**). A commit attempted past the ceiling requires a **re-mint** even if the hold is still live, capping a leaked-token window at the ceiling regardless of how far out the hold expires. Until the FED-AUTH-08 epoch store exists, **TTL is the only revocation bound that works**, so it must be genuinely short.

**FED-AUTH-11** 📋 (candidate) **Capture fires on the state change, not on a valid verify.** A within-TTL replay of a valid capability against an already-committed hold returns the **INV-06 idempotent success-echo** and must capture **nothing**. Payment capture (PAY-01) is triggered by the **first-time `HoldCommitted` WAL append** (a real mutation), never by "verify succeeded", so a within-TTL capability replay double-books nothing (AVAIL-07) and double-captures nothing. Test: same valid capability presented twice within TTL ⇒ one commit, one capture, the second a no-op echo with no second `PaymentIntent` capture.

**FED-AUTH-12** 📋 (candidate) **Re-mint enforces subset; no widening.** A single-hop re-mint MUST be a subset of the root on `{resource, TTL, price, depth}`; reject any widening. A Biscuit attenuation block that fails to nest within its parent's time/resource/amount bounds is rejected by the authorizer (monotonic attenuation). Unbounded sub-delegated tokens are rejected (FED-AUTH-03).

**Phasing (extends §7).** None of FED-AUTH-04..12 is built before **FED-08 + a real multi-hop-delegation requirement**. **P4a** (single operator): a **local** PASETO `use:cap` key, `audience = this node`, the FED-AUTH-08 edge fencing store, the FED-AUTH-09 wall+mono check: **no `.well-known` publication, no peer fetch, no Biscuit**; the dispatcher returns *unsupported-format* for Biscuit until triggered. **P4b** (cross-org, on FED-08): `.well-known/deltat-keys` publication, peer verification, cross-audience rejection. **Biscuit lights up only on the first real sub-delegation requirement** (NOT-05). The **next sprint is unchanged and format-independent**: P0a (PROTO-AUTH-08, kill the `wire.rs:72` connection-metadata tenant path) + HW-01/02/03/20 + AVAIL-07 + AVAIL-08. The entire capability/federation/Biscuit story needs **zero** `src/engine` change (PROTO-AUTH-00).

## Sharing & delegated grants

> **Status: recorded destination, not a mandate.** Folds SEC-02 ACL / SEC-03 hold-capability / FED-AUTH-03 delegation into one model: **CCAP (Compiled Capabilities)**, the authoritative home for how one principal lets a foreign person/app/AI do a scoped slice of their stuff. **Non-binding until FED-08** (a real second operator) **and** the first real multi-hop-delegation requirement, like section 9. Refines (does not override) PROTO-AUTH-05/06/08/10, FED-04/05, FED-AUTH-01..12, FED-09, SEC-03, AVAIL-07/08/11, HW-01/02/03+HW-20. **The next sprint is unchanged and grant-model-independent**: P0a (PROTO-AUTH-08) + HW-01/02/03/20 + AVAIL-07 + AVAIL-08, and none of this touches `src/engine` (PROTO-AUTH-00).
>
> **Fact-checked against HEAD (`feat/clock-seam`).** Nothing here is built: grep over `src/*.rs` for grant/capability/principal/biscuit/oauth/delegat returns nothing; `auth.rs:18` returns one shared password for any user; `wire.rs:72` derives the tenant from connection metadata and `get_or_create` lets a caller mint a tenant by naming it; `wire.rs:104-382` dispatches every `Command` with no authz; the commit path is `release_hold` (`wire.rs:176`) then `confirm_booking` (`wire.rs:188`), two separate kernel commands, a real TOCTOU (the orchestration is the demo client's, `demo/server.ts:88-97`; the kernel exposes no combined commit).

### SHR-00: One truth, two faces, kernel untouched

**SHR-00** 📋 **CCAP = a grant-registry ROW (the truth) + a minted capability (the wire).** Every grant is one durable edge registry row per home/tenant; the wire artifact is a Biscuit (FED-AUTH-04) minted FROM that row. Operators read/mutate **rows** (legible, `SELECT`-enumerable, auditable, instantly revocable); machines carry **capabilities** (offline, attenuable, sub-ms verify); the kernel sees neither; still `(Command, Ulid, now)` (PROTO-AUTH-00). The registry is exactly what registry-less ocap throws away (no table = no enumerate, no delete-to-revoke); CCAP keeps it because *show everyone with access* and *remove access NOW* are non-negotiable, at the cost of a cold-path MINT write only, never on the hot path (FED-AUTH-06). Row/capability/kernel cannot drift: the row is the only mint source, TTL bounds drift (FED-AUTH-10), an epoch bump (SHR-40) kills every copy.

**SHR-01** 📋 **Registry row: NEW DURABLE EDGE STATE, outside the WAL.** `{grant_id, grantee (principal_id | anon_link_id), rights_map, resource_selector, caps, on_behalf_of?, ttl, epoch_at_mint, issued_by, issued_at}`. PROTO-AUTH-00 forbids this in `src/engine` (absent from `model.rs`; MODEL-11 vocabulary frozen), so the registry log is outside the WAL, reconstructed on restart, **fail-closed** on kernel disagreement (like the FED-AUTH-08 fencing store). Grant changes are first-class NOTIFY-class edge events `{actor, oldRights, newRights}` (JMAP `ShareNotification`) so clients react live and revocation is audit-visible.

### SHR-10: The seven grant types (each maps to existing Commands, zero kernel change)

- **SHR-10a Anonymous free/busy share-link.** `{grantee=anon_link_id, role=freebusy-reader, selector=one resource, ttl, epoch}`, a degenerate single-block read-only capability in an unguessable URL with the token in the `#` **fragment** (W3C capability-URL guidance, out of logs/`Referer`). Maps to `SelectAvailability` **and** `SelectMultiAvailability`, **free spans only** (AVAIL-01, PROTO-AUTH-05, never `SelectHolds`, never hold ids/labels). Per-IP rate-limit **and** per-query cost-bound on **both** read shapes (cap window width + IN-list size, mirroring `MAX_IN_CLAUSE_IDS`) so it can't be scraped (SEC-05). Never a write/commit/payment path. Each link has its own `anon_link_id`-scoped epoch (SHR-40) so one rotates without breaking the others.
- **SHR-10b Identity grant (person|group|room|resource).** `{grantee=principal_id, role in {reader,booker,manager}, selector=resource-set | subtree-root}`. One first-class **Principal** over the resource tree (JMAP, a room books a room). Subtree grant: the edge walks `ResourceState.parent_id` to the granted root before dispatch, so the whole clinic/household is one row.
- **SHR-10c Book-on-behalf (attributable, not impersonating).** Adds `on_behalf_of=owner-U`; the capability carries actor=agent **and** user=U (FED-AUTH-03). On commit the edge dispatches as U but records a signed booking-receipt `{agent, user}`: the audit shows *agent booked on behalf of U* (Exchange send-on-behalf vs send-as, PROTO-AUTH-06). Depth-bounded (FED-AUTH-12).
- **SHR-10d App / AI OAuth consent.** App/AI is an OAuth 2.1 client; tap is **Resource Server only** (PROTO-AUTH-02). The validated bearer is not carried inward; the edge **exchanges** it for a scoped inward capability (PROTO-AUTH-06 structural no-passthrough). An AI agent's capability carries hard bounds as attenuation facts (budget ceiling, allowed categories, only within free windows, commit-TTL ceiling 120s, FED-AUTH-10). The agent **must hold before it commits**; payment settles on the hold `Ulid` (PAY-01). Re-verified every commit (PROTO-AUTH-03).
- **SHR-10e Reseller / channel (audience-bound venue→OTA).** `{grantee=OTA-principal, role=booker, selector=room-type pools (capacity-N), audience=OTA}`. **Audience-bound** (FED-AUTH-02): a cap minted for OTA-X is rejected if presented by OTA-Y. The venue stays single writer (FED-02); the OTA holds a scoped revocable cap, attenuable to sub-channels only narrower. LISTEN/NOTIFY (PROTO-12) bound to the authed tenant makes overselling structurally impossible (INV-01): One Room, Four Channels.
- **SHR-10f Time-boxed / one-time / ephemeral.** Either a row with `ttl=engagement-window` + epoch (contractor Q3 weekdays, concierge expires-1h), or a single-use commit-cap authorizing exactly one `CommitHold` on a specific `hold_id` (SEC-03), audience-bound, short TTL. Single-use enforced by FED-AUTH-11 (capture fires on the first `HoldCommitted` WAL append; a within-TTL replay is an idempotent no-op echo). The hold self-expires (AVAIL-11). Powers the pay-link / Ticketmaster-cart / no-show fix (VIS-08). Early-kill = epoch bump.

### SHR-20: Roles compile DOWN to one rights_map (no second enforcement table)

**SHR-20** 📋 **The enforcement unit is an orthogonal boolean `rights_map`, NOT an ordinal ladder** (JMAP `CalendarRights`, never Google's `none<freeBusy<reader<writer<owner`). A ladder forces *writer also sees private*, wrong for a booking engine where *see I'm busy but never WHAT* is the commonest external case. Vocabulary: `mayReadAvailability` (free spans only, its own cheap right), `mayReadBookings`, `mayReadHolds` (**owner-of-the-hold only**, SHR-21), `mayHold`, `mayBook` (+ `BatchInsertBookings` + commit), `mayBookOnBehalf` (actor!=user receipt), `mayShare` (mint sub-grants/attenuation blocks), `mayAdmin` (`Insert/Update/DeleteResource`, `InsertRule`, + the ACL).

**SHR-21** 📋 **`mayReadHolds` is scoped to the caller's OWN holds, never resource-wide.** `SelectHolds` (`queries.rs:253`) returns bare hold `Ulid`s + spans + `expires_at`, a near-authorizer for commit (SEC-03). Resource-wide hold visibility for every `booker` would let one booker enumerate **other** bookers' in-flight holds on a shared resource (the OTA, front desk, partner on the shared car), a cross-grantee leak and hold-id harvest. The edge **MUST** filter `get_holds` to the caller's grant before encoding; *owner-only* means owner-of-the-hold, not owner-of-the-resource. Resource-wide `SelectHolds` is a `mayAdmin` right, never in the default `booker` preset.

**SHR-22** 📋 **Four named ROLES are presets, a compiler front-end, never a stored table.** `freebusy-reader={mayReadAvailability}`; `reader=+{mayReadBookings}`; `booker=reader+{mayHold, mayBook, mayReadHolds}`; `manager=booker+{mayBookOnBehalf, mayShare, mayAdmin}`. Roles are the language humans/consent screens speak; they compile DOWN to the row's `rights_map` and from there to capability Datalog: **one source of truth** (row → capability), no second enforcement table to drift (DRY). A custom combination (e.g. `mayHold` without `mayBook`) has no role name and is a raw `rights_map`.

**SHR-23** 📋 **Selector + caps + eligibility ceiling.** Each row carries a **resource selector** (explicit `Ulid` list checked against the `Command`'s `resource_id` | subtree-root | `realm=tenant`) and **caps** (price ceiling, daily/total count, business-hours/window predicate, max-duration, capacity ceiling, single-use, commit-TTL <= 120s). An **eligibility ceiling** (MS-Graph `allowedRoles`/`isRemovable`) lets a resource or its parent cap which rights a principal **class** may ever receive (e.g. external/anon never exceed `mayReadAvailability`). Enforcement: SHR-24.

**SHR-24** 📋 **Eligibility ceiling + anti-escalation are VERIFIER-side facts over the FULL block chain: mint-time checks are necessary but not sufficient.** Offline multi-hop attenuation appends blocks the minting home never sees, and a ceiling tightened **after** mint must still reach an in-flight token. So: (a) the authorizer carries the current ceiling for the principal-class as a fact **sourced from the edge store** (like the fencing epoch, FED-AUTH-08); (b) every block's rights are a subset of **both** its parent (monotonic, FED-AUTH-12) **and** the live ceiling; (c) a tightening is bound to an epoch bump so it invalidates outstanding chains. Mint-time checks remain a cheap first gate.

**SHR-25** 📋 **Stateful caps reserve at HOLD time, against a durable edge accumulator.** `daily_cap`, total count, and budget are read-modify-write **accumulators**, not stateless predicates: N concurrent commits each pass a stateless commit-time check and overspend. A durable **per-grant accumulator store** (co-located with the fencing store, outside the kernel, reconstructed on restart) **reserves** budget/count **when the HOLD is placed** (charge at hold time, release on expiry/void, mirroring payment capture on the hold `Ulid`, PAY-01), so the commit-time check sees an already-reserved envelope and two concurrent commits can't both succeed. Stateless caps (price, window, max-duration) stay pure edge Datalog over `Command` fields.

### SHR-30. Delegation: opt-in, monotonic, depth-bounded, attributable

**SHR-30** 📋 **Delegation = a `mayShare` holder (or an OAuth/AI consent) mints a strictly-narrower sub-grant.** **Single-hop** (common): re-mint a tighter **flat** capability from the online home (one verify), a subset on `{resource, TTL, price, depth, rights}` (FED-AUTH-12, no widening). **Multi-hop offline** (user→agent→sub-agent, manager→report, venue→OTA→sub-channel): **attenuation blocks**, each strictly nested (monotonic), appended with zero issuer contact, depth/scope-bounded (FED-AUTH-03, cap 3 from measurement). Two orthogonal axes (FED-AUTH-07): delegation depth (block count, capped) and hop depth (cross-home), 0 by construction because every cross-home hop mints fresh (FED-AUTH-02 no passthrough).

**SHR-31** 📋 **Attenuable tokens are OPT-IN; the verifier enforces an absolute hard depth ceiling BEFORE running Datalog.** A multi-block capability never exists without an explicit `mayShare` grant in the consented scope (default = flat single-hop re-mint, FED-AUTH-07). The verifier rejects on **block count > MAX** before the `d` verifies: an absolute ceiling distinct from the tunable policy depth, so a holder can't feed a depth-10 chain to burn CPU before rejection (the deep-chain DoS vector; benchmark the real ruleset on the deploy target, TEST-11/HW-07, and verify-then-lock so a slow/rejecting verify never serializes unrelated commits).

**SHR-32** 📋 **Delegation is ATTRIBUTABLE, not impersonating.** The anti-escalation invariant (*you can only delegate what you hold, and only with `mayShare`*) is enforced at **both** mint (registry-side, cheap) **and** verify (Datalog, structural, SHR-24). The home records **both** agent and user U on every commit via the signed booking-receipt (PROTO-AUTH-06), unlike Google's impersonating delegation.

### SHR-40. Revocation: three composed mechanisms

**SHR-40** 📋 **Reconciling capability-TTL with operator remove-access-NOW**: registry-less ocap has only #3; the registry adds #1, the fencing store adds #2.

1. **Delete/mutate the registry ROW**: the operator-facing audit-visible action; stops all **future** mints for that grantee. Not sufficient alone: an in-flight capability is unaffected until it expires.
2. **Bump a fencing EPOCH**: the instant-revoke kill switch (FED-05/FED-AUTH-08). A per-tenant **edge store** holds the authoritative epochs the verifier reads (the token carries epoch/nonce as **claims**, the store holds the **truth**, comparing a token to its own field is circular). **Granularity is layered so the per-grantee use-cases are actually delivered, not just the resource-wide panic button:** a **per-grant** epoch (`grant_id → epoch`) kills exactly one grantee (*remove my friend*, *fire one contractor*) without disturbing co-grantees on a shared resource; a **per-link** epoch (`anon_link_id → epoch`) rotates one share-link without breaking the others; the **per-resource** epoch (`resource_id → epoch`) is the panic button. Each is one cached integer compare, no ACL lookup on the hot path. For offline-attenuated children, propagate the parent's revocation_id into each child block so a blocklist denies a whole subtree by one id; a child whose id was never captured is killed only by the parent-grant or resource epoch.
3. **Short TTL**, the always-on backstop (FED-AUTH-10): `commit-cap wall_deadline = now + min(120s ceiling, hold.expires_at - now)`. Until the epoch store exists TTL is the only bound that works, so it must be genuinely short, and is only sound on a monotonic-floored wall clock (FED-AUTH-09, HW-01/02/03+HW-20): a backward NTP step or post-restart wall jump otherwise makes an expired cap **and** an expired hold read as live. The HW-20 restart case (hold + cap both stay expired across a backward jump) is a hard prerequisite before any TTL'd capability ships.

**SHR-41** 📋 **Edge-store crash recovery has a pinned safe-failure DIRECTION.** Persist the epoch bump / row delete / accumulator reservation **before** the action they gate. On restart the kernel WAL replays to authoritative truth first, then edge stores reconcile: a hold the fencing store knows but the kernel doesn't => treat as **revoked** (fail closed); a committed hold the store missed => kernel wins, idempotent. A lost epoch bump fails toward *revoked*, a lost accumulator decrement toward *budget exhausted*. An HW-20-style partial-crash test (crash after epoch bump but before its log fsync => token stays revoked) is required.

### SHR-50: Why this is the power-AND-efficiency optimum

**SHR-50** 📋 **POWERFUL** because it is the only model simultaneously **legible** (read rows, revoke by deletion/epoch, enumerate with a `SELECT`) **and** forge-proof + federated + attenuable, no trade: the `rights_map` expresses splits a calendar can't (`mayReadAvailability` without `mayReadBookings`; *book the TV twice/day but never the car*; *OTA books room-type-A cap-N but sees no labels*); capacity-aware grants answer *at least k of N free together?* in one `SelectMultiAvailability` (FED-09 read is topology-free, so the same sweep composes across one home or twenty sovereign ones, VIS-03); attenuation gives true offline least-privilege delegation; auto-expiring holds (AVAIL-11) make *mid-checkout* real cross-everyone state (VIS-08, PAY-01). **EFFICIENT** because the hot path is one Ed25519 `verify_strict` (~28-41us) + integer/string compares against **edge** caches, zero DB lookup, zero network I/O, all before the per-resource write lock (verify-then-lock, FED-AUTH-06), vs a pure-ACL's identity-resolution + DB round-trip per check. The registry is touched only at mint (cold path), never per request. The kernel stays a pure `(Command, Ulid, now)` machine (PROTO-AUTH-00).

### SHR-90: Prerequisite or nothing (hard gates, in order)

**SHR-90** 📋 **No grant model is enforceable, and no commit-gating capability ships, before three gates land, in this order.** Until then the blast radius of *any* grant is **everything**, because the live system has no principal to grant **to** and no gate to enforce a grant **at**.

1. **P0a: PROTO-AUTH-08.** Bind the tenant to an authenticated principal; **delete** the `wire.rs:72` connection-metadata path (today attacker-chosen **and**, via `get_or_create`, lets a caller mint a tenant by naming it); reject if unbound; bind LISTEN/NOTIFY to the authed tenant. Strictly more restrictive than today. No regression. There is no principal to grant to until this lands.
2. **AVAIL-07: atomic single-lock `HoldCommitted`.** A signed commit-capability authorizes **who** may commit; it does **not** make the commit atomic. Today's only commit path is `release_hold` (`wire.rs:176`) then `confirm_booking` (`wire.rs:188`), two separate commands, a real transient-free-window TOCTOU (the orchestration is `demo/server.ts:88-97`; the kernel exposes no combined commit). Shipping any commit-cap before AVAIL-07 makes replay + race a **double-book**, not the FED-AUTH-11 idempotent echo. Test: two `CommitHold` racers + one independent `confirm` racer on one span => exactly one winner, no transient free window.
3. **HW-01/02/03 + HW-20: monotonic-floored expiry.** Expiry compares `CLOCK_REALTIME` (`clock.rs:25`), so a backward NTP step or post-restart wall jump makes an expired hold **and** an expired short-TTL capability read as live, voiding the leaked-token-dies-in-120s guarantee (FED-AUTH-09/10). Land the monotonic-floored check + HW-20 tests before any capability ships.

**SHR-91** 📋 **Standing rejections.** Macaroons / shared-HMAC stay rejected forever (cross-home forgeability, FED-AUTH-01); raw JWT (alg-confusion); UCAN on the hot path (same N-verify cost, drags DID/IPLD). Capability-URLs leak by design: confine to low-stakes free/busy reads, token in the `#` fragment, short TTL + per-link epoch, **never** a hold/commit/payment path (SHR-10a). The OAuth bearer→inward-capability exchange must be **structural** (the runner's dispatch entrypoint accepts only the verified inward capability type; the bearer has no constructor into `Command`), a property with a test that no bearer→dispatch-input constructor exists, not a code-review rule (PROTO-AUTH-06, lands with the PROTO-AUTH-11 runner).
